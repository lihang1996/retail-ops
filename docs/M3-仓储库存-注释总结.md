## 库存三维度管理

```
┌─────────────────────────────────────────────────┐
│                  stocks 表                       │
│                                                  │
│  total_qty = 100      (总库存/物理库存)         │
│    ├─ available_qty = 70   (可用库存)          │
│    └─ locked_qty = 30      (锁定库存)          │
│                                                  │
│  in_transit_qty = 0   (在途库存/预留)          │
│  version = 5          (乐观锁版本号)            │
└─────────────────────────────────────────────────┘

状态转换：
1. 入库：  total↑, available↑
2. 锁定：  available↓, locked↑
3. 解锁：  locked↓, available↑
4. 出库：  total↓, locked↓ (或 available↓)
```

## 乐观锁并发控制

### 原理

```sql
-- 1. 读取时获取 version
SELECT * FROM stocks WHERE stock_id = 'X'
-- 返回：{ stock_id: 'X', available_qty: 100, version: 3 }

-- 2. 更新时带上 version 条件
UPDATE stocks 
SET available_qty = 90, version = 4
WHERE stock_id = 'X' AND version = 3

-- 3. 检查 affected rows
-- affected = 1: 更新成功
-- affected = 0: version 已变，报并发冲突
```

### 并发场景示例

```
时间线：
t1: 事务 A 读取 stock (version=3, available_qty=100)
t2: 事务 B 读取 stock (version=3, available_qty=100)
t3: 事务 A 提交：UPDATE ... SET available_qty=90, version=4 WHERE version=3
    ✅ 成功，affected=1，version 变为 4
t4: 事务 B 提交：UPDATE ... SET available_qty=80, version=4 WHERE version=3
    ❌ 失败，affected=0 (version 已是 4，不匹配)
    → 报错：40900 库存并发冲突，请重试
```

### 错误处理

```javascript
// 前端重试逻辑
try {
  await api.post('/api/proj/stock/lock', { sku_id, qty })
} catch (error) {
  if (error.code === 40900) {
    // 并发冲突，自动重试
    await sleep(100)
    return retry()
  }
  throw error
}
```

### 1. 入库流程

```
┌─────────┐
│ 入库请求 │
│ qty=50  │
└────┬────┘
     │
     ▼
┌──────────────────────────────┐

┌──────────────────────────────┐
│ 2. updateStockWithVersion()  │
│    total_qty += 50           │
│    available_qty += 50       │
│    version += 1              │
└────┬─────────────────────────┘
     │
     ▼
┌──────────────────────────────┐
│ 3. upsertLocationQty()       │
│    更新库位级库存             │
└────┬─────────────────────────┘
     │
     ▼

```

### 2. 锁定流程

```
┌─────────┐
│ 锁定请求 │
│ qty=30  │
└────┬────┘
     │
     ▼
┌──────────────────────────────┐
│ 1. 检查 available_qty >= 30  │
│    不足则报错                 │
└────┬─────────────────────────┘
     │
     ▼
┌──────────────────────────────┐
│ 2. updateStockWithVersion()  │
│    available_qty -= 30       │

     │
     ▼

│    记录锁定流水               │
└──────────────────────────────┘
```

## 设计亮点

### 1. 乐观锁并发控制
通过 `version` 字段防止并发写冲突，无需数据库锁：

```javascript
// 读取时获取 version
const stock = await trx('stocks').where({ stock_id }).first()
// version = 3

// 更新时带上 version 条件
const affected = await trx('stocks')
  .where({ stock_id, version: 3 })
  .update({ available_qty: 90, version: 4 })

// affected=0 表示冲突，重试
if (!affected) throw new Error('并发冲突')
```

### 2. 库存三维度管理
清晰的库存状态划分：

```
total_qty      = 物理库存（真实存在的货物）
available_qty  = 可用库存（可销售的数量）
locked_qty     = 锁定库存（已预留待出库）
```

### 3. 懒创建模式
按需创建库存记录，节省存储空间：

```javascript
// 不预先创建所有 SKU-仓库组合
// 仅在首次操作时创建
const stock = await getOrCreateStock(trx, { skuId, warehouseId })
```

### 4. 库存红线
统一写入口，保证数据一致性：

```
Controller → Service (唯一入口) → Helper → Database
                ↓
         所有变更记录流水
```
